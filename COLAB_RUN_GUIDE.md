# Colab Run Guide

This guide is for running the project in Google Colab without changing the evaluation protocol or weakening model performance.

## 1. Prepare the project folder in Google Drive

Upload the whole project folder to Google Drive. The folder pointed to by PROJECT_ROOT must contain at least:

- requirements.txt
- src/
- digits4000_txt/
- challenge/
- digits_project_colab.ipynb

Recommended layout in Drive:

```text
MyDrive/
  CS5487 Course Project Code/
    requirements.txt
    src/
    digits4000_txt/
    challenge/
    digits_project_colab.ipynb
```

## 2. Open the notebook in Colab

Open [digits_project_colab.ipynb](digits_project_colab.ipynb) from Google Drive in Colab.

At the top of the notebook, keep or adjust these variables:

```python
USE_GOOGLE_DRIVE = True
PROJECT_ROOT = "/content/drive/MyDrive/CS5487 Course Project Code"
GRID_SEARCH_JOBS = 1
BATCH_PRESET = None
RUN_EXPERIMENTS = True
SELECTED_TRIAL_NAMES = None
SELECTED_MODEL_NAMES = None
RUN_NAME = None
COMBINE_RUN_NAMES = None
```

Meaning of each variable:

- USE_GOOGLE_DRIVE: mount Google Drive before running
- PROJECT_ROOT: the folder containing the full project
- GRID_SEARCH_JOBS: how many CPU workers GridSearchCV uses inside one model search
- BATCH_PRESET: apply a tested batch configuration for common resume workflows
- RUN_EXPERIMENTS: choose whether this notebook launch should start a new training run
- SELECTED_TRIAL_NAMES: run only part of the official trials when needed
- SELECTED_MODEL_NAMES: run only part of the model registry when needed
- RUN_NAME: write one batch into its own artifacts/runs/<name>/ directory
- COMBINE_RUN_NAMES: list of batch names to merge back into artifacts/results/

If your project folder is not directly under MyDrive, only change PROJECT_ROOT.

## 3. Run the notebook in order

Run the cells from top to bottom.

The notebook does the following:

1. Mount Google Drive and validate PROJECT_ROOT.
2. Install dependencies from requirements.txt.
3. Add src/ to Python path and configure runtime settings.
4. Check dataset shapes and official trial names.
5. Inspect current saved outputs and print completed versus pending `trial/model` pairs.
6. Run the configured experiment batch only when `RUN_EXPERIMENTS = True`.
7. Optionally combine finished batch runs back into canonical files under artifacts/results/.

Do not skip the data sanity cell. It is the fastest way to catch a wrong folder path.

## 4. How to reduce runtime without sacrificing model performance

The safe method is batching, not shrinking the search space.

Batching keeps the same models, preprocessing options, CV folds, and parameter grids. It only splits the work into smaller runs so Colab stays stable.

Each batch should have its own RUN_NAME. After all batches finish, set COMBINE_RUN_NAMES to the completed batch names and rerun the final results cell.

Recommended execution order:

### Batch A: lighter models

```python
BATCH_PRESET = "light"
```

### Batch B: heavier models

```python
BATCH_PRESET = "heavy"
```

If Batch B is still too slow, run one heavy model at a time:

```python
BATCH_PRESET = "rbf_only"
```

or

```python
BATCH_PRESET = "random_forest_only"
```

or

```python
BATCH_PRESET = "mlp_only"
```

You can also restrict trials temporarily while debugging:

```python
SELECTED_TRIAL_NAMES = ["trial_1"]
```

For the final reported results, switch both filters back to None and let all required runs finish.

After all batches are done, combine them with:

```python
RUN_EXPERIMENTS = False
COMBINE_RUN_NAMES = ["batch_light", "batch_heavy"]
```

The notebook will then rebuild the canonical summary files under artifacts/results/ without starting a new training run.

## 5. Where outputs come from

The outputs are generated by [src/digits_project/experiment.py](src/digits_project/experiment.py), which calls the shared reporting utilities and writes results under artifacts/.

Main output files:

- artifacts/runs/<run_name>/results/cv_leaderboard.csv
- artifacts/runs/<run_name>/results/final_selected_models.csv
- artifacts/runs/<run_name>/results/summary_by_model.csv
- artifacts/runs/<run_name>/results/email_summary.csv
- artifacts/runs/<run_name>/models/
- artifacts/results/: combined canonical outputs after COMBINE_RUN_NAMES is set

If a run stops midway, completed batch folders remain useful. This is why batching is the preferred Colab workflow.

## 6. Common problems

### FileNotFoundError for PROJECT_ROOT

Cause: PROJECT_ROOT does not point to the folder that contains requirements.txt and the dataset folders.

Fix: correct PROJECT_ROOT and rerun from the first code cell.

### Colab session uses too much CPU or memory

Cause: too many parallel workers or too many heavy models in one run.

Fix:

- keep GRID_SEARCH_JOBS at 1 or 2
- use SELECTED_MODEL_NAMES to run batches
- avoid opening multiple experiment notebooks at the same time

### Notebook runtime resets

Fix: reconnect the runtime, rerun the notebook from the dependency-install cell, read the completion summary cell, and continue only the missing batch.
